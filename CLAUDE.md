# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Antlers is a single-flake Nix monorepo of reusable packages and templates for Calamoose Labs / NKC. There is no application code. It targets `x86_64-linux` only (hardcoded). Layout:

- `flakes/` — package definitions as plain `callPackage`-able `package.nix` files (e.g. a wrapped `zed-editor`, a fixed `plex-desktop`, the `antlers` CLI). **Not** flakes themselves.
- `templates/` — `nix flake init`/`new` templates, registered in `templates/templates.nix`. Most are LaTeX/SC-IM document builders.
- `scripts/` — currently empty (placeholder).

**The root `flake.nix` is the single aggregator** — it exposes every package and template by attribute. Consumers grab one thing with `#<name>` (packages) or `-t #<name>` (templates). There is one branch (`master`); there are deliberately **no per-package branches** — with any `github:` ref Nix downloads the whole repo tarball regardless, so branches reduce nothing and only add merge maintenance. Everything assumes [direnv](https://direnv.net) + flakes: dirs with an `.envrc` (`use flake`) auto-load a dev shell (`direnv allow`).

## Common commands

```sh
# Work ON antlers
direnv allow              # or: nix develop   (root dev shell ships claude-code)
nix fmt                   # format with alejandra (the flake formatter)
nix flake check           # validate every output (run after edits)
nix flake show            # inspect the output tree

# Build / run a package
nix build .#zed-editor        # or .#plex-desktop ; output at ./result
nix run   .#zed-editor

# Scaffold from a template
nix flake init      -t .#<name>          # into cwd  (name ∈ templates/templates.nix)
nix flake new ./out -t .#<name>          # into a new dir

# Or use the `antlers` CLI (shorthand over all of the above — once it's on PATH)
antlers list                              # show every template + package (queried from the flake)
antlers new <name> [dir]                  # scaffold a template (dir defaults to ./<name>)
antlers init <name>                       # scaffold into the current directory
antlers build <package>                   # nix build .#<package>
antlers run <package>                     # nix run .#<package>
antlers shell                             # enter the antlers dev shell
# Targets github:CalamooseLabs/antlers by default; override with ANTLERS_REF.

# Build a document template's PDF (run inside the scaffolded/template dir)
nix build                                 # → ./result/main.pdf
```

Testing is minimal: the only automated test suite is **`checks.${system}.vibe-server-unit`** — offline Deno unit/integration tests under `flakes/vibe-server/app/test/` (`*_test.ts` + a zero-import `assert.ts`), run by `nix flake check` via `deno test` (covers `util`, `auth` cookie/rate-limit/password, `claude.ts` `extractLoginUrl`/`parseAuthStatus`/`mergeOnboarding`/`extractLoginError`, `sessions` `shQuote`/`substitute`, the pure path/name helpers in `paths_test.ts` (`normalizeAbs`/`withinRoot`/`sanitizeName`/`uniqueName`/`basenameOf`/`isLoopbackIp`), `diff.ts` `gitDiff` against a temp git repo, `term.ts` the VT/xterm log emulator in `term_test.ts` (SGR-strip, OSC drop, CR/CHA/CUP positioning, erase, LF-scroll + DECSTBM region, autowrap, alt-screen no-ops, chunk-split escapes + UTF-8, runaway-sequence bound), and `transcript.ts` the JSONL-transcript renderer in `transcript_test.ts` (`mangleProjectDir`, `summarizeToolInput` per tool, `renderRecord` for user/assistant/tool_use/tool_result + thinking-skip + bookkeeping-skip, `renderTranscript` parse-error-tolerance + blank-collapse + truncation)). Everything else (the other packages, the `vibe` launcher bash, all runtime behavior of the modules) is unverified beyond build-time gates: `deno compile` type-checks vibe-server/lanserver TS, `writeShellApplication` shellchecks the `vibe` launcher, and `nix flake check` + `nix fmt` (alejandra) validate evaluation/formatting. New TS belongs with a test; **keep tests import-free** (no `jsr:`/`npm:`/`@std` — use `app/test/assert.ts`) so they stay offline and the build's deno-cache FOD stays empty.

**Gotcha:** new `.nix` files must be `git add`-ed (commit not required) or flake evaluation can't see them and reports a misleading "path does not exist".

## Architecture

### Flake topology (single root aggregator)

The root `flake.nix` is the only flake exposing packages/templates. It:
- `callPackage`s each `flakes/<name>/package.nix` and exposes concrete derivations under `packages.${system}.{zed-editor, plex-desktop, antlers, default}`.
- exposes the **parameterized** Zed builder under `lib.${system}.mkZedWrapper` (a function, which is *not* allowed under `packages` — flake `packages.<system>.<name>` must be derivations).
- exposes `overlays.default` (so NixOS/home-manager can consume), `apps.${system}` (for `nix run`), `templates = import ./templates/templates.nix`, the root `devShells.default`, and `formatter` (alejandra).

Packages live in `package.nix` files (not sub-flakes) so the root flake reads them in-tree via `callPackage` — no `github:`-self-reference, no extra lockfiles, local edits visible immediately. **Do not** reintroduce a `flakes/flake.nix` or per-dir flakes; add new packages as `flakes/<name>/package.nix` and wire them into the root flake.

**The zed-editor wrapper** (`flakes/zed-editor/package.nix`) is the key reusable abstraction: a function `settings: derivation`. The resulting `zeditor` launcher writes the settings into a throwaway `XDG_CONFIG_HOME`/`XDG_DATA_HOME`, deep-merges the user's real `~/.config/zed/settings.json` via `jq -s '.[0] * .[1]'` (**right operand wins → the project's pinned settings take precedence**), copies the user's extensions/themes, and cleans up on exit. Get a concrete editor with `packages.zed-editor` (default settings) or customize via `lib.${system}.mkZedWrapper { … }`.

### The `antlers` CLI

`flakes/antlers/` packages a `writeShellApplication` named `antlers` that is a thin shorthand over the `nix` invocations against this flake — it does **not** reimplement anything. `package.nix` bakes a `defaultRef` (`github:CalamooseLabs/antlers`) into `antlers.sh` via `replaceStrings`/`readFile`; the user overrides it per-invocation with `ANTLERS_REF`. Subcommands: `list` (queries `#templates` and `#packages.<system>` with `nix eval --apply` + `jq` so it auto-discovers — nothing to hardcode), `new`/`init` (→ `nix flake new`/`init -t`), `build`/`run` (→ `nix build`/`run`), `shell` (→ `nix develop`), plus a hidden `completions <templates|packages|commands>` (bare newline-separated names, used only by the bash completion). Per-template commands like `create-lease` are **not** wrapped — they live in each scaffolded template's own dev shell.

**Bash completion** (`flakes/antlers/antlers-completion.bash`) makes `antlers <TAB>` complete subcommands, then `new`/`init` → live **template** names and `build`/`run` → live **package** names (and `new`'s `[dir]` arg → directories). The dynamic candidates are pulled from the flake via the hidden `antlers completions` helper — same auto-discovery as `list`, single source of truth — and cached under `$XDG_CACHE_HOME/antlers` (keyed by ref) so completion stays fast and never blocks: the cache is always served instantly and a `nix eval` refresh runs **only in the background** (when the cache is missing/empty or >1 day old), bounded by a `timeout` and serialized by a single-flight lock so a slow/offline eval or mashed TABs can't freeze the shell or pile up processes — the only cost is that the first completion before any cache exists yields no dynamic names (the next one does). `__DEFAULT_REF__` is substituted into both `antlers.sh` and the completion (the latter only for the cache key). The completion is **build-time gated** by `shellcheck` (it has no shebang, so it carries a `# shellcheck shell=bash` directive). It's installed via `installShellCompletion --cmd antlers --bash` → `$out/share/bash-completion/completions/antlers.bash` (the `.bash` suffix is intentional — bash-completion's lazy loader probes both `<cmd>` and `<cmd>.bash`). Because the loader resolves a command to its **real** path to find the adjacent completion, the wrapper derivation **copies** `bin/antlers` (not a symlink) so the binary and its completion share one store prefix. Needs bash-completion enabled on the host (NixOS default).

It is exposed three ways so it can land on a system: `packages.<system>.antlers`, `apps.<system>.antlers` (`nix run`), and `overlays.default.antlers`. To put it on a NixOS system, add the overlay and `environment.systemPackages = [ pkgs.antlers ];`, or reference `inputs.antlers.packages.${system}.antlers` directly. Editing the script is editing `flakes/antlers/antlers.sh` (a plain bash body — no shebang; `writeShellApplication` adds the shebang, `set -euo pipefail`, and a build-time shellcheck gate); the package is now a small `runCommandLocal` wrapper around that `writeShellApplication` that adds the completion. `git add` new files after creating, per the flake/git gotcha above.

### `vibe` (Claude Code launcher + browser session manager)

`vibe` is two halves split across **two** NixOS modules (each independently `enable`-able, usable alone, and integrating when both are imported):
- `nixosModules.vibe` (`import ./flakes/vibe/module.nix self`) → **`programs.vibe`** only (the launcher).
- `nixosModules.vibe-server` (`import ./flakes/vibe-server/module.nix self`) → **`services.vibe-server`** only (the web service). It builds the default per-session launcher with `mkVibeWrapper` from `../vibe/package.nix`; if the `vibe` module is also imported (`config.programs ? vibe`), `vibePackage` defaults to the `programs.vibe`-built launcher so sessions inherit the model/effort/permissions pins — otherwise it falls back to a default-config launcher (`mkVibeWrapper { remoteControl = true; }`). **Keep the vibe README about `programs.vibe` only and the vibe-server README about `services.vibe-server` only.**

- **`programs.vibe`** installs a configured `vibe` command. `flakes/vibe/package.nix` follows the **zed-editor wrapper pattern** — a `callPackage`-able *function* `cfg → writeShellApplication "vibe"` (exposed as `lib.<system>.mkVibeWrapper`; default-config derivation as `packages.<system>.vibe`). `vibe` serialises the pinned `{ model, effortLevel, ultracode, permissions, …extraSettings }` to a throwaway settings.json and runs `claude --settings …` (`VIBE_MODEL`/`VIBE_EFFORT`/`VIBE_ULTRACODE` env-override via jq). **ultracode** (`programs.vibe.ultracode` → settings.json `ultracode = true`) is a Claude Code toggle distinct from `effortLevel`: xhigh effort + dynamic multi-agent workflow orchestration, delivered via `--settings` like the other pins. **Permission mode** is the exception — `programs.vibe.permissionMode` (default `"auto"`) is delivered via the top-level **`claude --permission-mode <mode>` FLAG**, NOT settings.json, because a `defaultMode` in a `--settings` file is treated as project/local and `auto` from there is ignored; the flag is the reliable launch-time override and works for Remote Control sessions too (verified empirically — `auto` IS available in Remote Control, contrary to some docs; ineligible model/version silently falls back to `default`). `VIBE_PERMISSION_MODE` overrides per-run. `vibe --remote-control [name]` (or `VIBE_REMOTE_CONTROL=1`) adds the top-level `--remote-control [name]` **flag** to that same `claude --settings …` invocation so the session is driven from claude.ai / mobile. **Session name:** with no explicit name (positional arg / `VIBE_NAME` / configured `remoteControl.name`), vibe auto-generates `[<prefix>-]<repo>-<YYYYMMDD>` — `<repo>` = basename of the working dir's git toplevel (cwd fallback, sanitized to `[A-Za-z0-9_-]`), `<prefix>` from `remoteControl.prefix` / `VIBE_NAME_PREFIX` (builder param `namePrefix`). The name is now resolved **unconditionally** (not just in `--remote-control` mode) because it's also used for **self-registration**: if a `vibe-server` runs on the host, the launcher reads its discovery file (`${VIBE_SERVER_ENDPOINT:-/run/vibe/endpoint.json}`), POSTs its name/dir/pid to `/api/register` (loopback + token), background-heartbeats, and deregisters on exit — so a manually-run `vibe` shows up in the web UI (needs the bundled `curl`; all best-effort). To make the EXIT trap fire for deregistration, vibe no longer `exec`s `claude` when registered (or when a temp settings file needs cleanup); a single `cleanup()` handles the temp dir + heartbeat kill + deregister. Skip with `VIBE_NO_REGISTER=1`; server-spawned sessions carry `VIBE_MANAGED=1` and skip automatically. This is distinct from the server-side `services.vibe-server.sessionNamePrefix` (which the server prepends to its own generated `@NAME@` before passing it explicitly — that explicit name then bypasses the launcher's auto-gen unless you drop `@NAME@` from `sessionCommand`). **Key distinction (verified against claude-code 2.1.170): use the `--remote-control` FLAG on the main `claude` command, not the `claude remote-control` SUBCOMMAND.** The flag runs the normal interactive command with Remote Control enabled, so `--settings` (and thus the pinned model/effort/permissions) is fully honoured; the subcommand only accepts `--name`/`--permission-mode`, which is why the older path could not deliver model/effort. So the `programs.vibe.model`/`effort`/`permissions` pins now reach **both** interactive `vibe` use **and** the web-service (`services.vibe-server`) sessions (the remote client at claude.ai / mobile can still switch model client-side, but `--settings` sets the session default). **Subscription-first defaults** (vibe targets Claude Code Max/Team/Pro plans): `model` defaults to `opus[1m]` (latest Opus + 1M context — included on Max/Team, Pro draws usage credits), and `subscriptionAuth` (default true) drops a stray `ANTHROPIC_API_KEY` so sessions use the plan's OAuth login instead of silently billing the API (opt out with `subscriptionAuth = false` / `VIBE_API_KEY_AUTH=1`). **Presets** (`programs.vibe.presets.<name>`, builder param `presets`) — `vibe @<name>` starts a named preset: it `cd`s into the preset's **first** directory and passes the rest as `claude --add-dir` (one session, multiple accessible dirs), uses the preset's resolved pins (each null field inherits the top-level `programs.vibe.*`) via a per-preset settings.json + permission mode, optionally **checks out its `branch`** (`git checkout [-b]`, created from HEAD if absent, best-effort), and seeds the session name. The builder bakes a `vibe-presets.json` manifest (name → `{dir, addDirs, branch, settings, permissionMode}`) read by jq at runtime; the `@?*` parse happens before `--remote-control` parsing so `vibe @x --remote-control` composes. Presets also carry `branch`/`pushRemote`/`commitRequiresTouch`/`pushRequiresTouch` for vibe-server's Commit & Push. The resolver lives in `flakes/vibe/presets.nix` (`import ./presets.nix lib pcfg`, shared so both modules bake identical presets) and exposes `resolvedPresets`; per-preset assertions require ≥1 absolute directory and a valid branch name. **vibe-server consumes these presets — they SUPERSEDE the old `services.vibe-server.directories` (removed):** the service reads `config.programs.vibe.presets`, lists them in the UI, makes every preset directory writable, and spawns `vibe @<preset>`.

- **`services.vibe-server`** runs `vibe-server`, a Deno web service following the **lanserver pattern** (`deno compile`, FOD deno-cache, denort zip, nix-ld). The app is split into focused modules under `flakes/vibe-server/app/src/` (`main`, `config`, `auth`, `claude`, `sessions`, `presets`, `diff`, `commit`, `usage`, `sse`, `term`, `transcript`, `http`, `html`, `router`, `util`); **zero *external* imports** keeps the deno-cache FOD empty so the build stays offline — local `./*.ts` relative imports are fine (they download nothing), but do **not** add `jsr:`/`npm:`/`https:`/`@std/*`. The web UI: shared-password login (per-IP brute-force rate-limit; size-capped request bodies) — **optional**: when no `passwordFile` is configured (`services.vibe-server.passwordFile = null`, the default) the UI is passwordless (a public `GET /api/auth-mode` tells the page, `/api/login` then issues a cookie to anyone and the page auto-signs-in) → HMAC-signed cookie (`Secure` set only behind TLS, detected via `x-forwarded-proto`); lists the **presets** (`programs.vibe.presets`, which supersede the old `directories`); spawns a `vibe @<preset>` session per chosen preset (via the overridable `sessionCommand`, default `["${vibePackage}/bin/vibe" "@PRESET@" "--remote-control" "@NAME@"]`, `@PRESET@`→`@<name>` plus `@DIR@`/`@NAME@` substituted, cwd = the preset's first dir, launched under `setsid` for clean process-group kill **and a PTY via util-linux `script`** (`services.vibe-server.pty`, default true) — without a TTY, `claude --remote-control` auto-detects piped stdio and dies in headless `--print` mode (`Input must be provided … when using --print`); the PTY keeps it interactive (set `pty = false` only for a non-interactive `sessionCommand`). **PTY size:** `script`'s pipe-backed PTY reports **0×0**, so Claude/Ink falls back to an ~80×24 viewport — and since `claude --remote-control` is a full-screen alt-screen TUI that repaints its viewport in place (verified against real logs: ~34k cursor-homes, ~0 scrolls, and an empirically-confirmed clip to the terminal's row count), that 24-row clip is exactly why longer output looked "lost" (older rows were overwritten, never scrolled). So `sessions.ts` prepends `stty rows <ptyRows> cols <ptyCols>` to the `-c` command (set on the PTY's controlling tty before claude reads it; `services.vibe-server.ptyRows`/`ptyCols`, defaults 50×120, 0 = leave at default) — verified on a real host: claude positions its UI at the configured bottom row, so a taller PTY renders more of each message into the captured screen (keep ptyRows ≲120, the term.ts render-grid height). Output is captured to `/var/lib/vibe/logs/`, spawn env **allowlisted** — `services.vibe-server.extraEnv` extends it so stray secrets don't reach Claude Code or its browser-readable logs); lists/kills sessions; **types a one-off message into a running session's prompt** (`POST /api/sessions/:id/message {message}` → `sessions.ts` `sendInput` writes the cleaned text + a CR into the PTY's stdin, which `script` forwards to `claude` — the SAME channel the claude-auth code-paste uses; server-owned **running PTY** sessions only — external → 409, gone → 404, no live writer/not running → 409, like Kill; the session must hold a live stdin writer, so re-adopted-after-restart sessions can't (their `child`/stdin is gone). Sessions spawn with `stdin: "piped"` when `pty` is on (was `null`) so the writer exists; the writer is closed on exit. `cleanInput` collapses newlines→spaces (one prompt, not per-line submit) + strips control bytes (no TUI escape injection) + caps at 16 KiB; `/api/sessions` carries per-session `canInput` so the UI shows a **Message** button + modal only when eligible. This is a "type-and-Enter" affordance, NOT a streamed conversation — use Remote Control on claude.ai for that. **Whether `claude --remote-control` actually accepts injected local-PTY keystrokes into its prompt is unverified in-sandbox (no claude binary) and needs a real-host smoke test** (`cleanInput` is unit-tested in `sessions_test.ts`); the `\r`-submits-vs-`\n` choice may need tweaking against a live TUI. SSE-streams each session log read-only. **Two log sources, transcript-first** (`GET /api/sessions/:id/logs` + `…/logs/download`): by default the server serves **Claude's complete JSONL transcript** — the full, append-only conversation Claude writes at `$CLAUDE_CONFIG_DIR/projects/<mangled-cwd>/<sessionId>.jsonl` — rendered to readable text by `transcript.ts` (`renderTranscript`: user prompts `❯`, assistant text `●`, tool calls `⏵`, truncated tool results `⎿`; **thinking blocks and bookkeeping records skipped**; tool results capped). This is the fix for "longer messages get lost": unlike the terminal screen, the transcript genuinely *continues on* — every turn of the whole session, not just the last viewport. The session→file mapping is `sessions.ts` `getTranscriptPath` → `transcript.ts` `findTranscript` (newest `*.jsonl` in the **realpath-mangled** cwd dir — Claude names it from its symlink-resolved `process.cwd()`, so we `Deno.realPath` first — touched at/after the session's `startedAt`; **re-resolved each tick, never cached**; null before the first turn / for external sessions). **One self-upgrading SSE stream** (`sse.ts` `streamSessionLog`) serves both sources: it renders the transcript **incrementally** (a streaming `TextDecoder` + partial-line buffer parse only the NEW bytes per tick, so per-tick CPU is O(delta), not O(filesize)), and because it re-resolves every tick it (a) **upgrades terminal→transcript** when the first turn appears even if the log was opened pre-turn, and (b) **tracks a rolled transcript** (e.g. `/clear` starts a new `sessionId.jsonl`). Delivery is retried every tick (decoupled from the change-gate) so a snapshot skipped while a slow client was backpressured still goes out once it catches up. Reads are tail-bounded to `MAX_TRANSCRIPT_BYTES` (8 MiB) so a marathon transcript can't block the single-threaded event loop; the live snapshot is further tail-capped to 512 KiB at a line boundary (Download serves the full ≤8 MiB tail). The browser **replaces** its `<pre>` per snapshot. **Fallback to the terminal screen** when there's no transcript yet (fresh session) or via `?view=terminal` (which makes the resolver report none): the capture is raw PTY output from a full-screen TUI, so a naive ANSI-strip just concatenates out-of-order fragments; instead `term.ts` (a small zero-import VT100/xterm **grid emulator** — CSI/OSC/cursor/erase/scroll-region/alt-screen, robust across chunk boundaries, oversized grid + trailing-blank trim so it needs no PTY-size coupling) interprets the stream and the same loop emits the **screen snapshot whenever it changes**; the download (`?view=terminal`) renders from the **last ≤1 MiB** (via the shared `transcript.ts` `readTail`), `?raw=1` serves the raw bytes. **Exposure note:** the transcript reflects whatever Claude actually read/ran (tool inputs + truncated tool results), so unlike the `diff.ts` viewer — which deliberately excludes `.gitignore`'d files — it CAN surface a `Read` of a gitignored secret or a secret printed to command stdout; it's auth-gated, but passwordless mode (the default) hands a cookie to anyone who can reach the UI. **Empirically verified on a real host (claude-code 2.1.177):** claude repaints a fixed alt-screen viewport sized to the PTY (so the terminal view can only ever show one screenful — hence the transcript-first default), and the JSONL render was validated against real session transcripts; `term.ts`'s pure emulator and `transcript.ts`'s pure renderers are also unit-tested. It then serves a **per-session git diff** (`GET /api/sessions/:id/diff` → JSON `DiffResult`, shown in a responsive mobile+desktop **modal** with collapsible per-file cards) of the session's working directory — git runs **read-only** (no index/repo mutation) with a scrubbed env (no system/global/user config, `--no-ext-diff`/`--no-textconv` on **both** diff calls so a repo-local textconv/external-diff driver can't exec, `core.bigFileThreshold` to cap one blob's diff, `GIT_OPTIONAL_LOCKS=0` so it never blocks on `index.lock` while Claude writes), a fixed argv with `--` terminators, a per-call + aggregate timeout, and byte/file caps; it covers tracked changes (`git diff HEAD`, or `--cached` pre-first-commit) + untracked **non-ignored** files (`.gitignore`'d secrets never appear; binaries show as a content-less binary card; >256 KiB untracked files are skipped) and renders via `textContent` only (no XSS); not-a-repo and a clean tree are normal states, not errors; and surfaces a **login hyperlink** when a session's output contains a Claude/Anthropic OAuth URL (`SessionInfo.loginUrl`; the host set now includes `claude.com`, the `claude auth login` flow's host). It also shows a **live Claude usage panel** (module `usage.ts`, `services.vibe-server.usage.enable`, default on) — the same server-authoritative numbers the interactive `/usage` shows: the ~5h session window + weekly all-models/Sonnet/Opus windows + extra-usage credits, each a percent bar with a live reset countdown. The whole panel is **collapsible** (click the header caret; collapse state persists across the 20s re-render and reloads via `localStorage` key `vibe.usageCollapsed`). vibe-server reads the OAuth access token from `.credentials.json` **read-only** (never refreshing/rewriting it — that would race claude's own token rotation; an expired token just shows a "a running session will refresh it" note) and GETs Anthropic's **undocumented** `https://api.anthropic.com/api/oauth/usage` (exactly what the `claude` CLI calls — verified in the 2.1.177 bundle) with `Authorization: Bearer`, `anthropic-beta: oauth-2025-04-20`, and `User-Agent: claude-code/<installed version>` (memoized from `claude --version`; a wrong/absent UA drops into a punitive 429 bucket). The result is **cached + single-flighted**, refetched at most once per `usage.refreshInterval` (default 300s, floored at 180s in `usage.ts` since the endpoint is aggressively rate-limited) — every browser polls that one cached snapshot via the authed `GET /api/usage` and ticks the reset countdowns **locally** (so per-viewer load never hits the endpoint), and a transient 429/timeout keeps the last good snapshot visible, flagged `stale`. **Subscription/OAuth only** (no meaning under API-key billing — the panel then reports unavailable); needs no Nix/network change (the Deno build's `--allow-net` is already unrestricted). Pure `parseUsage`/`normResetsAt` are unit-tested in `usage_test.ts`, but **the live endpoint call is unverified in-sandbox (no outbound network / no real OAuth token) and needs a real-host smoke test.** **Optionally** (off by default, `services.vibe-server.commitPush.enable`) the UI gains a per-session **Commit & Push** button (module `commit.ts`) — the **only** path that *mutates* a repo (a remote analogue of the `gcommit` flow): it `git add -A`s the session's working tree, makes a **YubiKey/OpenPGP-signed** `git commit -S` (no AI trailer; identity/`signingkey` from the signing user's `~/.gitconfig`), then optionally `git push`es. The card **PIN** is typed in the web modal, written to a `0600` **tmpfs** file (`/run/vibe`, never argv/disk/captured-log), and fed to gpg via `-c gpg.program=<vibe-gpg wrapper>` + `VIBE_PIN_FILE` using `--pinentry-mode loopback --passphrase-file` for **one** signing attempt (never retried → can't walk the card toward its 3-strike lock), deleted in a `finally`. Unlike `diff.ts` it does **not** scrub the git env — it inherits the daemon env and overrides `HOME`/`GNUPGHOME` to the signing user's (so git finds identity + key and gpg finds the card) plus `GIT_TERMINAL_PROMPT=0` (a missing push credential fails fast, never hangs the daemon). **A touch can't be supplied from a browser**, so **per-preset** flags gate it: a preset's `commitRequiresTouch` hides the whole button, `pushRequiresTouch` drops the push step — the server **re-checks both (403)** against the session's preset and refuses **external** sessions (409, like Kill) + sessions whose preset is gone; `/api/sessions` carries per-session `canCommit`/`canPush`/`commitBranch` so the UI hides what's unavailable (cosmetic — the route is the gate). It signs **server-side**, so it requires running the service **as the login user** that owns the card (the default `vibe` user/root can't reach it — a warning fires), `allow-loopback-pinentry` in that user's `gpg-agent.conf`, and `pcscd`/the agent reachable (`loginctl enable-linger`); the module derives `gnupgHome`/`home` from `user`, adds `~/.gnupg` to `ReadWritePaths`, and relaxes `ProtectHome`. Each **preset's `branch`** (default null) pins the branch: null commits on the session's current branch (never switches), a name is checked out before each commit (created from HEAD if absent — `git checkout [-b]`, after a `git status --porcelain` "anything to commit?" pre-check so a clean tree never triggers a switch) and pushed with `--set-upstream` (remote = the preset's `pushRemote`, defaulting to `origin`); it's argv-validated (`validBranch`, no leading `-`). Endpoint `POST /api/sessions/:id/commit-push {message,pin,push}`. Pure helpers (`cleanMessage`/`cleanPin`/`commitArgs`/`pushArgs`/`canCommit`/`canPush`) are unit-tested in `commit_test.ts`; **the live card-signing handshake is unverified in-sandbox (no YubiKey) and needs a real-host smoke test.** The app also owns an **account-level Claude auth flow** (module `claude.ts`): the UI shows `claude auth status` as a banner and, when not logged in, *"Log in to Claude"* runs **`claude auth login --claudeai`** server-side under a PTY (`script`) with **piped stdin** — it scans the output for the `claude.com` OAuth URL (surfaced as a link), the user authenticates and pastes the code back, and the app writes `code\n` into the PTY so the OAuth login lands in the shared config dir; every subsequently-spawned session then inherits it (one login in-flight at a time, auto-killed after 10 min; endpoints `GET /api/claude-auth`, `POST/DELETE /api/claude-auth/login`, `POST /api/claude-auth/code`, all cookie-gated). Empirically verified (claude-code 2.1.170): login = URL-on-`claude.com` + paste-back code over a PTY; `claude auth status --json` is non-interactive (exit 0 logged in / exit 1 not, JSON either way). To stop a fresh service user's sessions blocking on Claude Code's first-run **theme picker** and **workspace-trust** dialog, the app seeds `$CLAUDE_CONFIG_DIR/.claude.json` with `hasCompletedOnboarding` + `theme` (`services.vibe-server.claudeTheme`, default `"dark"`) + per-directory `hasTrustDialogAccepted`/`hasCompletedProjectOnboarding` (merging, never clobbering) — at startup for every preset directory and before each spawn for that session's preset dirs (toggle with `services.vibe-server.seedClaudeOnboarding`, default true). The module now **always** exports `CLAUDE_CONFIG_DIR` (→ `claudeConfigDir` if set, else `<stateDir>/.claude`) so the login, the seed, and the sessions all read/write the same deterministic location (`.claude.json`/`.credentials.json` live INSIDE that dir). Note `--allow-write` is compile-scoped to `/var/lib/vibe,/run/vibe`, so the Deno-side seed only works when the config dir is under the state dir (the default); a custom `claudeConfigDir` outside it relies on the pre-seed / UI-login subprocess paths instead. Launch targets are **config-defined presets** — there is no runtime filesystem browser / project creation (that was removed when presets superseded `directories`, along with `browseRoot`/`projectsDir`/`newProjectTemplate`). Every preset directory is added to the systemd `ReadWritePaths` so sessions can write (else `ProtectSystem=strict` makes them read-only). The only authed preset endpoint is `GET /api/presets`. A **manually-run `vibe`** on the host also appears: when vibe-server runs it drops a discovery file `/run/vibe/endpoint.json` (URL + per-process token, mode 0644) and the launcher self-registers via **`POST /api/register`** (gated by a **loopback peer + the token**, exempt from `requireTLS`), heartbeats (`PUT`), and deregisters (`DELETE`) on exit; such **external** sessions (`SessionInfo.external`) are listed with diff but **no captured-log tail and no Kill** (output lives in the user's terminal), are retired ~90s after heartbeats stop (not via `/proc` — `ProtectProc=invisible` hides cross-user pids), and are **not** snapshotted. Server-spawned sessions set `VIBE_MANAGED=1` so they skip self-registration. **Server-spawned sessions survive a vibe-server restart**: the running set is snapshotted to `stateDir/sessions.json` and re-adopted on boot (these have no `ChildProcess` handle, so a reaper polls `/proc` to retire ones that exited) — Remote Control sessions therefore aren't orphaned. An unauthenticated `/healthz` is exposed for liveness probes. Other knobs: `requireTLS` (reject non-TLS requests with 426, except `/healthz`), `sessionNamePrefix`, `maxLogBytes` (per-session log size cap — appends stop past it, no rotation so the live SSE tail keeps working), and `localNetworkSubnets6` (IPv6 firewall rules); the UI adds copy-session-name, uptime, confirm-before-kill, a `terminating` state, and per-session log metadata + download (`/api/sessions/:id/logs/download`). The `vibe` launcher itself has `--help` / `--show-config` (print the pinned settings + `claude auth status`). The browser is a **lifecycle manager** — you actually interact with a session through Claude Code Remote Control.

Unlike lanserver's strict sandbox, the unit uses a static `User`/`Group`, `StateDirectory=vibe`, `ReadWritePaths = ["/var/lib/vibe"] ++ directories`, and **auto-relaxes `ProtectHome`** (→ `false`, with a `warnings` notice) when any configured dir is under `/home`; never set `MemoryDenyWriteExecute` (breaks V8's JIT). It also applies broad systemd hardening that is safe for V8 (`UMask=0077`, `ProtectProc=invisible`, `ProtectKernel*`, `RestrictNamespaces`, `LockPersonality`, `RestrictAddressFamilies=AF_INET/AF_INET6/AF_UNIX/AF_NETLINK`, …) but **no `SystemCallFilter`** (also breaks the JIT). Module `assertions` reject non-absolute directory paths and names outside `[A-Za-z0-9_-]+` (which the web UI rejects anyway). The service user must be authenticated to Claude — for subscription plans (Max/Team/Pro) the recommended path is a pre-seeded `claudeConfigDir` (→ `CLAUDE_CONFIG_DIR`) holding an OAuth `/login`; an `environmentFile` (`ANTHROPIC_API_KEY=…`) is the API-key alternative (and then requires `programs.vibe.subscriptionAuth = false`, since the wrapper otherwise drops the key to keep sessions on the plan). `passwordFile`/`environmentFile` are read at runtime, never copied to the store. Note `templates/vibe-shell/` is the dev-shell template new UI-created projects are scaffolded from — it takes antlers as an input and puts `vibe` (via `mkVibeWrapper`) in its dev shell.

### Templates consuming antlers

The dev-shell templates pull the shared Zed wrapper by taking antlers as a flake input and calling the builder:

```nix
# flake.nix
inputs.antlers = { url = "github:CalamooseLabs/antlers"; inputs.nixpkgs.follows = "nixpkgs"; };
# shell.nix
(inputs.antlers.lib.x86_64-linux.mkZedWrapper zedSettings)
```

A scaffolded template ends up in a *separate* repo, so it must reference antlers via `github:` (not a relative path). Pin to a tag/rev for reproducibility.

### LaTeX document templates

`tex-editor`, `nkc-report`, `nkc-farmland-lease`, `nkc-lease-amendment` share a structure:

- `flake.nix` → `devShells.default` (a configured Zed + `texlab`/`alejandra`/`nixd`/`nil`) and `packages.default = pkgs.callPackage ./build.nix {}`.
- `build.nix` is a `stdenv.mkDerivation` that runs `latexmk -interaction=nonstopmode -outdir=build -pdf ./src/main.tex` under `texliveFull` and installs `build/main.pdf` to `$out`. (Use `-outdir=build` and copy `build/main.pdf` — a bare `-auxdir` puts the PDF in the aux dir under pdflatex's emulation, which breaks `cp main.pdf`.) So `nix build` → `result/main.pdf`.
- `src/main.tex` `\input`/`\subimport`s `style(s)`, `variables`, and `content`.

**Modular document tree** (most developed in `nkc-farmland-lease`): a recursive tree of directories, each with three files:
- `variables.tex` — an inclusion boolean (e.g. `\newboolean{includeArticleI}`) plus that node's content vars.
- `main.tex` — `\subimport{.}{variables}` then `\ifthenelse{\boolean{includeX}}{ \section{…} \subimport{.}{content} }{ \section{\omitText} … }` — toggling the boolean includes or stubs out the whole subtree.
- `content.tex` — the prose, which itself `\subimport`s child nodes (`article_* → section_* → subsection_* → subsubsection_*`).

Add a node by scaffolding from the template's own nested registry (`templates/<doc-template>/templates/default.nix`):
```sh
cd templates/<doc-template>
nix flake new <path> -t .#article   # or .#section / .#subsection / .#subsubsection
```

`nkc-report` is the simpler variant — flat `src/content.tex` with `\newtoggle` flags (`confidential`, `showdraft`, `showcomments`) for watermarks/headers, and no nested `templates/` dir.

### The Company, Inc. corporate documents (`thecompanyinc-*`) + `flakes/legal`

A family of 10 corporate-document templates (`thecompanyinc-annual-board-memo`, `-annual-shareholder-memo`, `-authentication-record-book`, `-memorandum-board`, `-memorandum-shareholders`, `-bill-of-sale`, `-ip-license`, `-promissory-note`, `-bylaws-amendment`, `-shareholder-agreement-amendment`) that all share **one** canonical style and **one** wizard, exposed from **`flakes/legal/`** (wired into the root flake like the other packages):

- **`packages.thecompanyinc-style`** — a derivation carrying the single canonical `thecompanyinc-style.tex` (Century Schoolbook, generalized from the master-lease style: `\section`→"ARTICLE \<Roman\>", run-in dotted subsections, defined-term machinery, `\exhibithead`/`\schedulehead`, `\sigline`/`\sigblock`/`\sigpair`/`\fillin`, `zref` footer, `hyperref` last). Editing this one file restyles every corporate doc.
- **`lib.mkLegalDoc { src; pname?; mainTex? }`** — the shared `latexmk`/`texliveFull` PDF builder; it puts the style dir on `TEXINPUTS`, so a doc's `src/main.tex` loads it with `\input{thecompanyinc-style}` (then `\subimport{./src}{defined_terms}` / `{variables}` / `{content}`). No per-doc `build.nix`.
- **`lib.docWizard`** (packaged as `thecompanyinc-doc-wizard`) — the shared **`create-doc`/`edit-doc`** commands. The DRY engine lives in `flakes/legal/wizard/` (`render_core.py` = generic formatters + a manifest-driven `derive()`; `create_doc.py` = the wizard). It reads each template's own **`scripts/manifest.py`** (`TITLE`, `INPUTS` [typed fill-ins: text/choice/count/floor/sqft/years/money/money_simple/percent/percent_legal/date/address; `firm=True` seeds `settings.json`, `blank=True` renders a `\fillin` rule], optional `NODES` tree, optional `DEFINED_TERMS`, optional `SUMMARY_FIELDS`), so **one** engine drives all 10. Same UX as `create-lease` (`settings.json` firm defaults > `doc.json` per-instance answers; `--defaults`/`--no-build`/`--save-settings`/`--scaffold`). Most of these docs are flat (`NODES=[]`) — the wizard just fills `src/variables.tex`; the agreements can use the node tree.

Each template dir is **thin**: `flake.nix` (`packages.default = antlers.lib.${system}.mkLegalDoc { src = ./.; }`), `shell.nix` (adds `antlersLib.docWizard` + sets `TEXINPUTS` so texlab resolves the style), `scripts/manifest.py`, `src/{main,content}.tex` (+ generated `variables.tex`/`defined_terms.tex`), `settings.json` (The Company, Inc. firm defaults).

**Legal-folder instances/records** live at `~/02 - Areas/The Company, Inc./legal/<Doc Name>/` — each a self-contained flake that consumes antlers via a **`path:`** input (`antlers.url = "path:/home/hub/01%20-%20Projects/calamooselabs/antlers"`; switch to `github:CalamooseLabs/antlers` + a pinned rev once pushed) and builds its PDF with `mkLegalDoc`. The reusable-doc folders are filled instances of the templates (wizard included); `Bylaws`, `Shareholder Agreement` (+ its Schedule A promissory note), and `Executive Summary` are **static records** (no wizard — `src/variables.tex` holds the real values directly).

### spreadsheet-pdf template

A different pipeline: `build.nix` drives `sc-im` headlessly via an `expect` heredoc to export `src/main.sc` to Markdown, then `pandoc` renders `main.pdf`. (The heredoc terminator is column-0 *after* Nix `''` strips the common indent — leave it indented in-source.)

## Conventions

- Canonical nixpkgs: `github:NixOS/nixpkgs/nixos-unstable`. Owner casing: `NixOS` and `CalamooseLabs`.
- Format with `nix fmt` / `alejandra` before committing.
- New flake inputs should `inputs.nixpkgs.follows = "nixpkgs"` to avoid duplicate nixpkgs.
- Each template dir carries its own `.envrc` (`use flake`) and a `.gitignore` covering `.direnv/` and `result`.
